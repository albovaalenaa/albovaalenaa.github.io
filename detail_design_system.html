Дизайн-система Kokoc Group

B2B, B2C
Design System
с 2024 — н.в.
Ситуация. Когда я пришла в Kokoc Group, экосистема из пяти продуктов — Платформа Kokoc Group, внутренняя HRM-система, Карьерный сайт, Лендинг Kokoc Group и CRM — держалась на разрозненном UI Kit.

Задача. Выстроить полноценную дизайн-систему — с токенами, библиотекой компонентов и синхронизацией с кодовой базой через Storybook — так, чтобы она стала единым источником правды для дизайнеров и разработчиков сразу во всех пяти продуктах.
Цель
Унифицировать UI пяти продуктов экосистемы, наладить дизайн-код синхронизацию
Продукты в периметре
Платформа Kokoc Group, HRM, Карьерный сайт, Лендинг, CRM
Моя роль
Продуктовый дизайнер, автор и владелец дизайн-системы
Инструменты
Figma (Variables, Component Properties), Storybook

1. Аудит существующего UI

Начала с инвентаризации: прошла по всем пяти продуктам и собрала скриншоты каждого повторяющегося элемента интерфейса — кнопок, полей ввода, карточек, типографики. Это сразу обнаружило масштаб проблемы: например, только оттенков «основного красного» цвета в интерфейсах нашлось больше десятка, а отступы между одинаковыми по смыслу блоками отличались в разных продуктах на глаз.
Пример находки аудита — разные оттенки одного и того же «фирменного красного» в разных продуктах

2. Токены как единый источник правды

На основе аудита собрала токен-архитектуру — цвет, типографика, отступы, радиусы, тени — платформонезависимую, не завязанную на конкретный продукт. Каждый токен получил осмысленное имя по назначению (например, --accent-main, а не red-3), чтобы дизайнер и разработчик понимали не только значение, но и где токен уместно использовать.
Токены цвета — единая палитра вместо десятка похожих оттенков
Интуитивно понятные переменные
Единая типографическая система
Единообразие использования в интерфейсе

3. Библиотека компонентов

Пересобрала ключевые компоненты на токенах — с вариантами и состояниями (default, hover, active, disabled, error), по принципам атомарного дизайна: от простых атомов (кнопка, поле ввода) до готовых блоков, которые можно переиспользовать в любом из пяти продуктов без доработки с нуля.
Состояния кнопки на токенах дизайн-системы

4. Документация и правила именования

До этого документации не существовало вовсе — новый дизайнер или разработчик мог узнать, как использовать компонент, только спросив коллегу. Для каждого компонента описала: назначение и сценарии использования, уникальное имя (единое для Figma и кода, чтобы дизайнер и разработчик всегда говорили об одном и том же), правила и ограничения (когда компонент уместен, а когда нет), примеры на реальных экранах.
Пример пункта из документации
 

5. Синхронизация с кодом через Storybook

Дизайн-система без синхронизации с кодом — это просто красивый Figma-файл, который начинает расходиться с реальным продуктом уже через пару спринтов. Поэтому вместе с разработчиками перенесла токены и компоненты в Storybook: изменение токена в одном месте распространяется на все продукты и на дизайн, и на код одновременно — без ручной синхронизации вручную по каждому продукту.

6. Роллаут на пять продуктов

Внедряла систему поэтапно, начиная с Платформы Kokoc Group как самого нагруженного продукта, затем — CRM, HRM, Карьерный сайт и Лендинг. На каждом продукте проверяла, не ломает ли миграция существующие сценарии, и донастраивала токены под специфику продукта там, где это было оправданно (не жертвуя единообразием).
